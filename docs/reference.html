Overview

Getting started

A lightweight, declarative framework for multi-page sites and web apps. HTML declares what happens; JavaScript defines how. No build step, no dependencies.

Mental model

Behaviour is bound to HTML elements via data-* attributes. One event listener on document handles everything. When you scan the HTML, it is immediately clear which element does what — without hunting through JS files.

<button data-click="openDialog(confirm)">Delete</button>
<button data-click="saveItem|closeDialog(confirm)">Confirm</button>
Overview

Load order

All scripts require defer or placement before </body>. project.js must come last.

Default — bundled. brio.js contains all core modules (built via npm run build):

<link rel="stylesheet" href="brio.css">
<script src="brio.js" defer></script>
<script src="project.js" defer></script>

Alternative — modular files (vendoring src/*.js into your project):

<script src="utils.js"    defer></script>
<script src="actions.js"  defer></script>
<script src="binding.js"  defer></script>
<script src="dialogs.js"  defer></script>
<script src="feedback.js" defer></script>
<script src="project.js"  defer></script>

Core

Binding actions

Core event types click, change, input, submit are always registered. Additional types can be added with registerEvent().

Syntax

<!-- Single action -->
<button data-click="myAction">…</button>

<!-- With a target argument -->
<button data-click="myAction(someId)">…</button>

<!-- Pipe-separated: runs both in order -->
<button data-click="saveItem|closeDialog(confirm)">…</button>

<!-- On change (select, checkbox, radio) -->
<select data-change="myAction">…</select>

<!-- On input — fires on every keystroke (live search, counters) -->
<input data-input="myAction" type="text">

<!-- On submit -->
<form data-submit="myAction">…</form>

Action function signature

function myAction(el, event, target) {
  // el     — the element that triggered the action (or null from URL)
  // event  — the native DOM event (or null from URL)
  // target — the string inside parentheses, or null
}

Registering extra event types

// Call before injecting elements that use this event type
registerEvent('dragstart');
registerEvent('dragover');

No-refresh append + template (reactive)

Use BRIO_API.applyResponse() to test the same state/append response shape your data-fetch endpoint returns.

    Keyboard shortcuts

    Add data-key to any element. Uses event.key strings. Shortcuts are ignored while the user is typing in a field.

    Syntax

    <!-- Single key -->
<button data-key="n" data-click="newItem">New</button>

<!-- Chord — modifiers always in Ctrl › Alt › Shift › Meta order -->
<button data-key="Ctrl+s"       data-click="save">Save</button>
<button data-key="Ctrl+Shift+z"  data-click="redo">Redo</button>
<button data-key="Escape"        data-click="closeDialog">Close</button>
    Modifier order matters. Chords must be written in Ctrl › Alt › Shift › Meta order — this is the order buildChord produces internally. Ctrl+Alt+s matches; Alt+Ctrl+s does not.
    Letter keys are case-sensitive — use lowercase: "n" not "N". Special keys keep their event.key casing: Escape, Enter, ArrowDown.

    Press N to open the modeless dialog (focus outside a text field).

    URL trigger

    Any action can be triggered by adding an ?action= parameter to the URL. Evaluated on DOMContentLoaded.

    https://example.com?action=openDialog(settings)
https://example.com?action=myAction(someId)

    Built-in actions

    Two shorthands are built in and need no function definition.

    <!-- Navigate to a URL -->
<a data-click="url:https://example.com">…</a>

<!-- Reload the page -->
<button data-click="refresh">…</button>
    Note: url: is checked before pipe-splitting, so URLs that happen to contain a | character are handled safely.

    Dialogs

    Manages modal and modeless dialogs. Focus is trapped inside modal dialogs; focus returns to the opener on close.

    Required HTML structure

  <button data-el="close-button" data-click="closeDialog(confirm)">Cancel</button>

</dialog>

    Stacking & layering

    Brio keeps dialogs visible by default with baseline z-index and supports stack flows. In this demo, open stack flow, then open a nested dialog from inside it, and close back to the previous one.

    Focus & Escape

    On open, focus moves to the first visible focusable element (skipping the close button), or to [data-autofocus] if present. On close, focus returns to the element that opened the dialog.

    Escape and backdrop dismiss are blocked unless an enabled [data-el=close-button] exists, which allows required-answer dialogs. Use role="alertdialog" for these cases.

    <!-- Force focus to a specific element -->
<textarea data-autofocus></textarea>

<!-- Prevent reopen when a second modal opens over this one -->
<dialog id="alert" data-reopen="false">

    Accessible names

    addDialogTitleIds() runs automatically at init. It assigns id="dialogId-title" to the first <h2> inside each dialog (skipping those that already have an id) so aria-labelledby can reference it. A console warning is logged for any dialog with no accessible name at all.

    Feedback & forms

    Inline messages for client-side and server-side feedback. Also the primary form validation system — no separate library needed.

    HTML patterns

    <!-- Field-level: aria-describedby wired automatically -->
<input id="email" type="email">
<span data-message data-for="email" hidden></span>

<!-- Form-level: scoped to nearest form -->
<form>
  <div data-message hidden></div>
  …
</form>

<!-- Non-form feedback: nearest sibling -->
<button data-click="addToCart">Add to cart</button>
<span data-message hidden></span>

<!-- Custom scope -->
<div data-feedback-scope>
  <div data-message hidden></div>
  …
</div>

    JavaScript API

    FunctionDescription
    showMessage(target, msg, type)Show a message. target is a field id string or triggering element. Types: error (default), success, info, warning
    readServerMessages(el, data)Distribute a parsed server response. Also calls enableForm automatically when success is false auto
    clearMessages(el)Clear all messages in the nearest form or [data-feedback-scope]

    Form locking

    actions.js calls disableForm automatically on every submit. readServerMessages calls enableForm automatically when success is false. Call enableForm manually if you handle errors outside readServerMessages.

    async function submitContact(el, event) {
  event.preventDefault();
  clearMessages(el);
  try {
    const data = await getData('/api/contact', true);
    readServerMessages(el, data); // enableForm called automatically if !data.success
    if (data.success) closeDialog(null, null, 'contact');
  } catch (e) {
    showMessage(el, 'Something went wrong.', 'error');
    enableForm(el.closest('form')?.id); // call manually in catch
  }
}

    Server response contract

    {
  "success": true,
  "messages": [
    { "field": "email", "message": "Please enter a valid email.", "type": "error" },
    { "field": null,    "message": "Something went wrong.",          "type": "error" }
  ]
}

    field: "id" routes to [data-message][data-for="id"]. field: null routes to the nearest form-level [data-message].

    Styling messages

    [data-message][data-type="error"]   { color: var(--color-error); }
[data-message][data-type="success"] { color: var(--color-success); }
[data-message][data-type="info"]    { color: var(--color-info); }
[data-message][data-type="warning"] { color: var(--color-warning); }

    Kirby CMS — Uniform helper

    function formResponse($form) {
  $messages = [];
  foreach ($form->errors() as $field => $errors) {
    foreach ($errors as $message) {
      $messages[] = ['field' => $field, 'message' => $message, 'type' => 'error'];
    }
  }
  return Response::json([
    'success'  => $form->success(),
    'messages' => $messages
  ], $form->success() ? 200 : 400);
}
    Components

    Dynamic fields

    Show, hide, enable, or disable elements based on other field values. No JavaScript needed. Re-evaluated on every change and submit event.

    Attributes

    AttributeEffect when condition is met
    data-show-whenRemoves hidden
    data-hide-whenAdds hidden
    data-enable-whenRemoves disabled
    data-disable-whenAdds disabled

    Condition syntax

    SyntaxMeaning
    fieldId==valueField equals value
    fieldId!=valueField does not equal value
    cond1||cond2OR — applied if ANY condition is met
    cond1&&cond2AND — applied if ALL conditions are met
    Note: || and && cannot be mixed in a single expression.

    Examples

    <!-- Show when plan = pro OR enterprise -->
<div data-show-when="plan==pro||plan==enterprise" hidden>…</div>

<!-- Show when plan = enterprise AND seats = 10 -->
<div data-show-when="plan==enterprise&&seats==10" hidden>…</div>

<!-- Hide when status is not active -->
<div data-hide-when="status!=active">…</div>

<!-- Disable when plan = free -->
<select data-disable-when="plan==free">…</select>

    Dynamic text

    Update visible text in response to field input or action results. Re-evaluated on every change, input, and submit event.

    Source syntax

    SyntaxReads from
    #idDOM element — .value for inputs/selects/textareas, .textContent for others
    object.propertyJS scope — dot-notation path resolved from window
    {{placeholders}}Template — detected when value contains {{. Mix both source types freely.

    Declarative

    <!-- Mirror a field value -->
<span data-bind="#field-id"></span>

<!-- Read from a JS object (translations, config) -->
<span data-bind="translations.saveButton"></span>

<!-- Template — mix #id and dot.notation -->
<span data-bind="{{#qty}} × {{pricing.currency}}{{#price}}"></span>

    Disclosure toggle

    Toggle visibility of any element. aria-expanded and aria-controls are managed automatically.

    <!-- Explicit target -->
<button data-click="toggle(faq-1)" aria-expanded="false">Question</button>
<div id="faq-1" hidden>Answer</div>

<!-- Implicit next sibling — id auto-generated if absent -->
<button data-click="toggle" aria-expanded="false">Show more</button>
<div hidden>More content</div>
    Add aria-expanded="false" to the trigger to declare the correct initial ARIA state before any interaction — toggle keeps it in sync from that point on.

    Writing action functions

    Define functions in project.js. Any function in global scope is immediately available as a data-* value — no registration needed.

    // Simple synchronous action
function saveItem(el, event, target) {
  // ...
}

// Async action with server feedback
async function bookSlot(el, event, target) {
  clearMessages(el);
  try {
    const data = await getData('/api/book/' + target, true);
    readServerMessages(el, data); // also calls enableForm if !data.success
    if (data.success) closeDialog(null, null, 'booking');
  } catch (e) {
    showMessage(el, 'Something went wrong. Please try again.', 'error');
    enableForm(el.closest('form')?.id);
  }
}

    Accessibility

    Dialogs

    On open, focus moves into the dialog. On close, focus returns to the opener. Native modal dialogs use browser top-layer behavior from showModal(), including background interaction blocking and focus containment.

    Escape closes modal dialogs — blocked when no enabled [data-el=close-button] is present. Use role="alertdialog" for dialogs that require a decision.

    Feedback messages

    [data-message] regions are announced automatically: errors via role="alert", all other types via aria-live="polite". Field-level messages are wired to their input via aria-describedby when showMessage is called — no manual wiring needed.

    Dynamic fields

    data-show-when / data-hide-when handle visual state only. For form-field-driven disclosure, add aria-controls="target-id" to the controlling element manually. If the revealed content is informational (not just a new input), also add aria-live="polite" to the revealed region.

    Keyboard shortcuts

    Letter keys in data-key are case-sensitive — use lowercase. Chords must be in Ctrl › Alt › Shift › Meta order. Shortcuts are ignored while focus is inside a text field.

Brio — API simplification
Developer experience

Reducing boilerplate in the HTML API

Six concrete opportunities — some confirmed wins, some worth a discussion. Each reduces the number of things a developer has to remember or write to get correct behaviour.

01

Combine data-message + data-for

Confirmed win
High — affects every form in every project

Your observation is correct. data-message is a valueless marker — it never holds a value. When it appears alongside data-for, both attributes together say "this is the message region for field X." Collapsing them into a single attribute is strictly cleaner and still leaves the form-level pattern (data-message alone) untouched.

Field-level message region
− <span data-message data-for="email" hidden></span>
+ <span data-message-for="email" hidden></span>
Form-level message region — unchanged
<div data-message hidden></div>

The JS change is small: resolveMessageRegion needs to query [data-message-for="id"] instead of [data-message][data-for="id"], and clearMessages needs to select both [data-message] and [data-message-for].

02

Remove data-el="close-button" entirely

Confirmed win
High — every dialog has one, developers currently need both attributes

Close buttons currently require two attributes: data-click="closeDialog" (to trigger close) and data-el="close-button" (so the framework knows whether Escape and dim-click should close the dialog). These two always appear on the same element and always mean the same thing. The framework can infer the second from the first — if there's an enabled data-click="closeDialog" button in the dialog, that IS the close button.

Close button
− <button data-click="closeDialog" data-el="close-button">Close</button>
+ <button data-click="closeDialog">Close</button>

The JS change is a one-line selector swap: replace [data-el=close-button]:not([disabled]) with [data-click="closeDialog"]:not([disabled]) in the two places it's checked (Escape handler and clickDimLayer). Also replace the same selector in autoFocus, which currently skips the close button when looking for the first focusable element.

03

Native <dialog> + aria-labelledby by default

Confirmed win
High — every dialog, removes two attributes and one manual step

Everything inside the dialog portal is a dialog — the framework already knows this. And addDialogTitleIds() already finds the <h2> inside each dialog and assigns it an id like settings-title, but then stops. The developer still has to go back to the dialog element and manually write aria-labelledby="settings-title". The function already has everything it needs to do this step automatically.

Before — developer writes
− <dialog id="settings" aria-labelledby="settings-title">
After — developer writes
+ <dialog id="settings">

Current direction: use native <dialog> directly and keep addDialogTitleIds() focused on assigning heading ids/labels when missing. No portal role injection needed.

04

Keep framework hooks namespaced (data-brio)

Confirmed win
Medium — clarity and safety, low effort

Framework hook attributes should stay explicitly namespaced. data-brio is clear, unambiguous, and avoids collisions with other tooling conventions.

Naming direction
− <div data-el="page-content">
− <div data-el="dialog-portal">
− <div data-el="dim-layer" data-click="clickDimLayer">
+ <div data-brio="page-content">
+ <div data-brio="dialog-portal">
+ <div data-brio="dim-layer" data-click="clickDimLayer">

This is a naming-level choice for internal hooks. It does not change the dialog API for consumers.

05

Native backdrop policy for <dialog>

Current behavior
Medium — removes custom overlay wiring and keeps behavior explicit

In the native <dialog> model there is no separate dim-layer element to wire. Backdrop behavior comes from native dialog mechanics plus Brio's dismissal policy.

Dismiss behavior
− Custom dim-layer click handling in HTML
+ Native <dialog> backdrop + Brio dismiss policy

The practical result is less boilerplate and fewer integration mistakes, while retaining policy control through the close-button rule.

06

Keep data-reopen="false" for stacked-dialog opt-out

Confirmed win
Low — niche, but clear in intent

In the current native dialog implementation, Brio keeps data-reopen="false" as the explicit opt-out for reopen behavior when stacked dialogs are used.

Dialog that should not reopen after a sub-dialog closes
+ <dialog id="settings" data-reopen="false">

JS behavior: reopen is disabled when data-reopen="false" is present.

Change Type Before After
Field message Combine data-message data-for="x" data-message-for="x"
Close button marker Remove data-el="close-button" Inferred from data-click="closeDialog"
Dialog role + label Auto <dialog id="x" aria-labelledby="x-title"> Native markup + optional heading id helper
Framework hook attribute Rename data-el="..." data-brio="..."
Backdrop dismiss Auto Custom dim-layer click handling Native backdrop + Brio policy
Reopen opt-out Keep data-reopen="false" data-reopen="false" (unchanged)

All changes except #5 are non-breaking relative to each other and can be applied in one pass. If any existing project code has been written against these APIs, a migration note in BACKLOG.md is recommended.

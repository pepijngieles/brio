Brio — LLM adoption analysis
Strategic analysis

Would an LLM use Brio?

Assuming tabs, accordions, and other missing patterns are implemented — what stands between Brio and genuine LLM adoption? And what would it take to change that?

Verdict

Not by default. Conditionally yes, if properly set up. An LLM encountering Brio cold — no docs in context, no instructions — will not use it. It will reach for Alpine.js, vanilla JS, or whatever framework dominates its training data. But an LLM given Brio's docs in context, especially in a well-structured format, is well-positioned to use Brio correctly. The pattern is learnable, small, and consistent. The problem is not capability — it's discoverability and context.

Why it won't happen by default

The training data problem is the whole game. When an LLM generates code, it works from patterns it has seen thousands of times. React, Alpine.js, vanilla JS event listeners, Bootstrap modals — these appear across millions of repositories, tutorials, and Stack Overflow answers. Brio appears in none of them. The LLM has no instinct for it, no examples to draw from, no muscle memory. Even if asked to "build a modal," it will generate what it knows.

This is not fixable by improving the framework. It is only fixable by getting Brio's patterns into the LLM's context window at the moment of code generation. The docs have to travel with the project.

This is actually the same problem every new framework faces — but established frameworks solved it by accumulating years of training data. Brio has to solve it differently: by being intentionally designed for inference-time adoption.

What works in Brio's favour

Once the docs are in context, Brio is surprisingly well-suited to LLM code generation. Several of its design decisions — made for human developers — turn out to be equally good for machines.

Brio advantages for LLMs
  • ✓ Declarative pattern is deterministic — data-click="fn(target)" has one correct interpretation
  • ✓ Small API — maybe 20 attributes and a handful of functions. An LLM can hold the whole thing in context
  • ✓ No build step — LLM doesn't need to generate webpack config, package.json, tsconfig
  • ✓ Consistent conventions — the pattern generalises. Once an LLM knows one Brio attribute, it can extrapolate
  • ✓ Readable generated HTML — LLM output is auditable; attributes signal intent
  • ✓ Function names as actions — data-click="openDialog(settings)" reads like plain English to an LLM
Alpine.js advantages for LLMs
  • ✓ Millions of training examples in the LLM's weights already
  • ✓ Self-contained — logic and markup in one place, easier to generate as a single unit
  • ✓ Inline expressions — LLM can write x-show="open" without needing a separate JS file
  • ✓ No scaffolding requirements — no portal, no dim layer, no page-content wrapper
  • ✓ LLM already knows which JS it can use — full inline JS expressions

The deeper advantage: Brio's separation of HTML and JS — which can feel like extra work to a human — actually maps well to how LLMs naturally generate code. An LLM can write the HTML structure first, then write the corresponding functions, with clear one-to-one correspondence between data-click="saveItem" and function saveItem(el, event, target) {}. The handshake is explicit.

The four real blockers

These are not theoretical — they are specific failure modes an LLM will hit when trying to use Brio without sufficient context.

1
The invisible scaffolding problem

Dialogs now require native <dialog id="..."> markup plus a matching action target. An LLM that uses a plain <div> or mismatched id will create dialog-like HTML that Brio cannot open. This is still a likely failure mode without docs in context, but the markup contract is now much smaller and fully native.

2
The function signature problem

Every action function must have the signature function myAction(el, event, target) {}. An LLM that guesses will probably write function myAction() {} or function myAction(event) {} — both common in vanilla JS. The first misses the target argument; the second gets el when it expects event. Both break silently. The LLM has no equivalent in its training data to pattern-match from, so it has to be explicitly told.

3
The "what functions exist" problem

Brio exposes a set of built-in functions — openDialog, closeDialog, toggle, showMessage, readServerMessages, etc. An LLM that doesn't know these exist will invent its own versions, or use Alpine-style patterns. It needs to know what's already available before it decides what to write. The docs list these functions, but discoverability still depends on delivering that reference in the LLM context.

4
The server contract problem

The readServerMessages() function expects a specific JSON shape: { success: bool, messages: [{ field, message, type }] }. An LLM generating a form handler will probably use its own convention — returning { error: "..." } or { status: "ok" } — and the feedback system will silently receive the response and do nothing. This is a cross-file contract that the LLM has to know about when generating both the frontend HTML and whatever backend endpoint it's wiring up.

The two-file coordination issue

This deserves its own section because it is structurally different from the problems above. With Alpine, an LLM writes everything in one place — the HTML attribute contains the logic. With Brio, the LLM must generate two coordinated artefacts: the HTML attribute, and the JS function in project.js. These must match exactly.

What the LLM must generate and keep in sync

<button data-click="saveItem(item-123)">Save</button>

// In project.js — must know this is needed, and know the signature
function saveItem(el, event, target) {
  // target === 'item-123' — must know this is how target arrives
  // el === the button element
  // event === the click event
}

For simple generation tasks this is fine — the LLM can write both. The risk is in partial context: if an LLM is asked to "add a save button" to existing HTML without being given the project.js file, it will generate the HTML attribute correctly, but won't know whether saveItem already exists or needs to be created. It may silently generate duplicate or conflicting functions.

The fix is documentation, not framework change. The convention "always show both sides of the HTML/JS pair in every example" makes this easy. The LLM learns to think of them as a unit.

What needs to change

Most of these are documentation and distribution decisions, not runtime changes.

Priority 1 — Context delivery
An LLM system prompt snippet
A single dense Markdown file — under 2000 tokens — that covers: load order, all data-attributes, function signature, built-in functions, dialog scaffolding, server contract. Designed to be pasted into a system prompt or dropped into a .cursorrules / CLAUDE.md file. This is the single highest-leverage deliverable. Every Brio project that includes it gets LLM support for free.
Priority 1 — Docs format
A Markdown version of the docs + llms.txt
The docs HTML is excellent for humans; a markdown companion remains better for LLM ingestion. A DOCS.md companion — same content, plain Markdown — combined with a llms.txt file on the Brio site lets coding assistants (Cursor, Windsurf, Claude, Copilot) load the right docs automatically. This is now an established convention with real adoption at Anthropic, Vercel, Cloudflare.
Priority 2 — Example completeness
Every example shows both sides
Every code example in the docs should show the HTML attribute AND the JS function it calls, in the same block. Currently some examples show only the HTML side. An LLM learning Brio from the docs will miss the connection if it only ever sees one half of the pair.
Priority 2 — Scaffolding
A canonical starter template
A complete, minimal index.html that has the correct load order and a canonical native <dialog> example — pre-wired. When an LLM is asked to "start a new Brio project," this is what it should output verbatim. It eliminates the invisible scaffolding problem entirely.
Priority 3 — Discoverability
Built-in function reference table
A compact table in the docs (and the system prompt snippet) listing every built-in action by name — openDialog, toggle, showMessage, etc. — with a one-line description of each. Prevents the LLM from inventing things Brio already handles.
Priority 3 — Server contract
The JSON contract as a copy-paste block
A clearly labelled, copy-paste-ready JSON example of the server response format — in the docs, in the starter template comments, and in the system prompt snippet. Every LLM generating a backend endpoint needs to see this before it invents its own format.

The honest comparison

Right now, if you ask an LLM to "build a contact form with a confirmation modal," here is what happens:

With Alpine.js — LLM has everything it needs in its weights
<div x-data="{ open: false }">
  <button @click="open = true">Submit</button>
  <div x-show="open">
    <p>Confirmed!</p>
    <button @click="open = false">Close</button>
  </div>
</div>
// No separate JS file needed. LLM generates this confidently, correctly.
With Brio — LLM has nothing in its weights, must guess
// LLM likely generates some of these correctly, some incorrectly:
<button data-click="openDialog(confirm)">Submit</button>

// Probably missing — LLM doesn't know this is required:
<dialog id="confirm">...</dialog>

// still required: id must match data-click="openDialog(confirm)"
With Brio + system prompt snippet — LLM has what it needs
// Generates correctly, confidently, completely
<button data-click="openDialog(confirm)">Submit</button>

<dialog id="confirm">
  <h2>Confirmed!</h2>
  <button data-el="close-button" data-click="closeDialog(confirm)">Close</button>
</dialog>

Bottom line

Brio's design is well-suited to LLM code generation. It is small, consistent, and explicit. An LLM that knows the pattern produces clean, correct, readable output — arguably cleaner than Alpine because the logic stays in JS where an LLM can reason about it, and the HTML stays declarative where a human can read it.

The remaining gap is mostly contextual. The LLM needs to be handed the right information at the right moment. That means: a system prompt snippet, a Markdown docs file, and a starter template. These three assets create reliable default behavior in AI-assisted workflows.

The opportunity is real. The trend in LLM-assisted development is toward smaller, more focused tools — not away from them. An LLM working within a Brio project, given the system prompt snippet, will likely produce better results than an LLM working with Alpine, because Brio's constraints reduce the surface area for mistakes. The framework does the accessibility work. The server contract is defined. The scaffolding is fixed. There are fewer decisions for the LLM to make, and fewer of them wrong.

The single most important thing to build: a brio.md file — under 2000 tokens — designed to be dropped into any AI coding assistant's context. When a developer adds that file to their project, every subsequent LLM interaction with Brio code becomes correct by default. That file is more valuable than any documentation page, because it travels with the code.
